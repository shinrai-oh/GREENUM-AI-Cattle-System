# 产品需求文档 (PRD) - 活体肉牛超声波肉质评估系统



| **文档版本** | **修改日期** | **修改人**    | **备注**                                           |
| ------------ | ------------ | ------------- | -------------------------------------------------- |
| V1.0         | 2025-11-06   | 产品经理 (AI) | 初始版本创建，明确前后端分离、MySQL 及模拟数据逻辑 |



## 1. 引言 (Introduction)





### 1.1 项目背景



目前，肉牛的肉质评级（如大理石花纹）大多在屠宰后进行，这是一种滞后的评估方式。超声波技术允许在肉牛活体状态下，通过测量背膘厚度、眼肌面积和肌内脂肪含量（IMF）等关键指标，非侵入性地预测其肉质。

然而，这些宝贵的测量数据往往分散保存在B超设备、Excel表格或纸质记录中，缺乏一个系统化的平台来进行数据汇集、个体追踪、群体分析和育种决策。



### 1.2 产品目标



本系统（V1.0）旨在为肉牛养殖场和育种中心提供一个中心化的数据管理平台。

- **目标1（数据采集）：** 提供一个标准化的界面，用于录入单头肉牛的超声波测量数据。
- **目标2（数据管理）：K** 使用 MySQL 数据库，安全、持久地存储每头牛的身份信息和多次测量历史。
- **目标3（数据模拟）：** 暂时使用**固定的模拟规则（Mock Logic）**，根据录入的指标（如IMF）自动生成一个模拟的“肉质评级”，为前端提供即时反馈。
- **目标4（数据展示）：** 提供清晰的报表和个体追踪视图，帮助管理者进行饲喂和销售决策。
- **目标5（技术基础）：** 采用前后端分离架构，为未来（V2.0）无缝接入真实的 AI 预测模型打下坚实基础。



## 2. 用户角色 (User Personas)



| **角色**              | **核心诉求**                                                 |
| --------------------- | ------------------------------------------------------------ |
| **超声波操作员/兽医** | 在牛舍现场，需要快速、准确地为特定牛只录入测量数据（可能通过平板电脑）。 |
| **牛场管理者**        | 查看整个牛群的平均肉质水平；识别哪些牛只已达到上市标准；评估饲喂方案的效果。 |
| **育种专家**          | 追踪特定种公牛/母牛后代的肉质表现（眼肌面积、IMF）；需要历史数据进行遗传评估。 |
| **系统管理员**        | 管理用户账户、角色权限、备份数据库。                         |



## 3. 系统架构与技术栈约束



- **架构:** B/S 架构，前后端分离。
- **前端 (FE):** 现代 Web 框架（如 Vue.js, React）。通过 RESTful API 与后端通信。
- **后端 (BE):** 现代 Web 框架（如 Spring Boot, Django, Node.js）。负责业务逻辑、API 接口、安全认证和数据持久化。
- **数据库:** **MySQL** (版本 5.7+ 或 8.0+)。
- **认证:** Token-Based Authentication (如 JWT)。



## 4. 功能需求 (Functional Requirements)





### 4.1 模块一：用户与权限管理



- **F1.1 用户登录：** 提供用户名/密码登录界面。
- **F1.2 角色管理：**
  - **管理员(Admin):** 拥有所有权限，可创建/管理用户。
  - **管理者(Manager):** 可查看所有报表、牛只数据，可录入/修改数据。
  - **操作员(Operator):** 仅可录入测量数据、查看自己录入的数据。



### 4.2 模块二：牛群档案管理



- **F2.1 牛只信息 CRUD：**
  - **创建/录入：** 核心字段必须包含：`耳标号 (EarTagID)` (唯一键)、`出生日期`、`品种`、`性别`、`父系耳标号`、`母系耳标号`、`所属牛舍/分组`。
  - **查询：** 支持按耳标号、品种、牛舍等多维度模糊搜索。
- **F2.2 牛群/分组管理：**
  - 管理者可创建不同的“分组”（如：“育肥A组”、“后备母牛群”），方便进行对比分析。
  - 支持将牛只批量分配到不同分组。



### 4.3 模块三：超声波数据录入 (核心功能)



- **F3.1 测量数据录入界面：**
  - **步骤1:** 操作员输入或选择 `耳标号` 来锁定目标牛只。
  - **步骤2:** 系统显示该牛只的基本信息（用于核对）。
  - **步骤3:** 操作员填写本次测量数据：
    - `测量日期` (必填)
    - `操作员姓名` (自动填充当前用户)
    - `背膘厚度 (Backfat Thickness)` (单位: mm)
    - `眼肌面积 (Ribeye Area)` (单位: cm²)
    - `肌内脂肪含量 (IMF)` (单位: %)
    - `眼肌高度 (Eye Muscle Height)` (单位: cm)
    - `眼肌宽度 (Eye Muscle Width)` (单位: cm)
    - `备注` (可选, 如 B 超图像编号)
- **F3.2 录入确认：**
  - 点击“保存”后，前端将数据 POST 至后端。
  - **后端触发：** 后端在将数据存入 MySQL *之前*，必须调用“**8. 模拟评估逻辑 (Mock Logic)**”模块。
  - **数据存储：** 将 F3.1 的录入数据 + F8.1 生成的 `模拟评级` 结果，一同存入`measurements` (测量记录表)。



### 4.4 模块四：数据查询与报表



- **F4.1 个体追踪报告：**
  - 输入 `耳标号` 查询。
  - **视图1:** 显示该牛只的最新一次测量数据和“模拟评级”。
  - **视图2:** 以图表（折线图）形式，展示该牛只**历次**的 `背膘厚度`、`眼肌面积`、`IMF` 变化趋势。
- **F4.2 群体对比报表：**
  - 用户可选择一个或多个“分组”（F2.2 中创建的）。
  - 以数据表格形式，展示所选分组牛群的各项指标**平均值**（平均IMF、平均眼肌面积等）。
- **F4.3 数据导出：**
  - 支持将 F4.1 和 F4.2 的数据导出为 `.csv` 或 Excel 文件。



## 5. 高阶 API 接口定义 (V1.0)



为指导前后端分离开发，后端需提供以下核心 API (RESTful):

| **动词** | **路径**                                 | **描述**                                           |
| -------- | ---------------------------------------- | -------------------------------------------------- |
| `POST`   | `/api/v1/auth/login`                     | 用户登录，获取 Token                               |
| `GET`    | `/api/v1/cattle`                         | 获取牛只列表（支持分页和搜索）                     |
| `POST`   | `/api/v1/cattle`                         | 新增一头牛只档案                                   |
| `GET`    | `/api/v1/cattle/{earTagId}`              | 获取特定牛只的详细档案                             |
| `PUT`    | `/api/v1/cattle/{earTagId}`              | 更新牛只档案                                       |
| `GET`    | `/api/v1/cattle/{earTagId}/measurements` | 获取特定牛只的**所有**历史测量记录                 |
| `POST`   | `/api/v1/measurements`                   | **(核心)** 提交一次新的超声波测量数据              |
| `GET`    | `/api/v1/reports/group`                  | 获取群体对比报表数据（通过 query params 指定分组） |



## 6. MySQL 数据库设计 (V1.0)



以下是 V1.0 必需的核心数据表结构：

- **`users`** (用户表)
  - `id` (INT, PK, AI)
  - `username` (VARCHAR, UNIQUE)
  - `password_hash` (VARCHAR)
  - `role` (ENUM('admin', 'manager', 'operator'))
- **`cattle_profiles`** (牛只档案表)
  - `id` (INT, PK, AI)
  - `ear_tag_id` (VARCHAR, UNIQUE, NOT NULL) - *核心业务键*
  - `birth_date` (DATE)
  - `breed` (VARCHAR) - *品种*
  - `sex` (ENUM('male', 'female'))
  - `sire_ear_tag_id` (VARCHAR) - *父系耳标*
  - `dam_ear_tag_id` (VARCHAR) - *母系耳标*
  - `group_id` (INT, FK -> `cattle_groups.id`) - *所属分组*
- **`cattle_groups`** (牛只分组表)
  - `id` (INT, PK, AI)
  - `group_name` (VARCHAR, NOT NULL)
  - `description` (TEXT)
- **`measurements`** (测量记录表) - *核心数据表*
  - `id` (INT, PK, AI)
  - `cattle_profile_id` (INT, FK -> `cattle_profiles.id`) - *关联牛只*
  - `user_id` (INT, FK -> `users.id`) - *操作员*
  - `measurement_date` (DATETIME, NOT NULL)
  - `backfat_thickness` (DECIMAL(5,2)) - *背膘 (mm)*
  - `ribeye_area` (DECIMAL(5,2)) - *眼肌面积 (cm²)*
  - `intramuscular_fat_imf` (DECIMAL(5,2)) - *肌内脂肪 (%)*
  - `ribeye_height` (DECIMAL(5,2)) - *眼肌高度 (cm)*
  - `ribeye_width` (DECIMAL(5,2)) - *眼肌宽度 (cm)*
  - `notes` (TEXT)
  - `**simulated_grade**` (VARCHAR) - ***存储模拟评级结果\***



## 8. 模拟评估逻辑 (Mock Logic) - [V1.0 临时替代方案]



此模块是 V1.0 的核心约束，用于替代 AI 算法。它将在后端被实现为一个内部服务或函数。

- **F8.1 模拟评级函数:**

  - **触发:** 当后端收到 `POST /api/v1/measurements` 请求时，在数据存入数据库前调用此函数。

  - **输入:** 本次测量的 `intramuscular_fat_imf` (IMF 值)。

  - **输出:** 一个评级字符串 ( `simulated_grade` )。

  - **模拟规则（示例）：**

    Java

    ```
    // 这是一个后端 Java/Python/Node.js 逻辑示例
    String calculateMockGrade(Double imf) {
        if (imf == null) {
            return "N/A";
        }
        if (imf >= 6.0) {
            return "Prime+ (A5)"; // 模拟最高等级
        } else if (imf >= 4.5) {
            return "Prime (A4)";
        } else if (imf >= 3.0) {
            return "Choice+ (A3)";
        } else if (imf >= 2.0) {
            return "Choice (A2)";
        } else {
            return "Standard"; // 模拟标准以下
        }
    }
    ```

  - **结果:** 此函数返回的字符串（如 "Prime (A4)"）将被存入 `measurements` 表的 `simulated_grade` 字段。前端在查询时直接显示此结果即可。
